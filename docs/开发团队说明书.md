## 二、角色详述

### 2.1 Product Owner（你）

| 维度 | 说明 |
|------|------|
| **身份** | 你本人，唯一的需求来源和验收权威 |
| **职责** | 提出业务需求、排定优先级、验收交付物、方向决策 |
| **不做的** | 写代码、调 Bug、管 Git、装环境、技术决策 |
| **参与节奏** | Sprint 首日提目标，Sprint 末日验收，中间可随时查看进度或改方向 |

### 2.2 Scrum Master + 架构师 + 安全审计（硅基先锋）

| 维度 | 说明 |
|------|------|
| **身份** | 主 Agent，你的直接对话对象 |
| **核心职责** | |
| — Scrum Master | 任务拆解、子代理派单、进度追踪、障碍移除 |
| — 架构师 | 技术方案评审、架构决策、代码一致性审查、技术债管理 |
| — 安全审计 | Sprint 末安全 checklist（数据加密、权限校验、SQL注入、XSS） |
| **Code Review 流程** | 子代理产出 → Scrum Master 审查架构/安全/一致性 → 通过后提交 Git |
| **增强 Skill** | proactive-agent（主动预警）、self-improving-agent（错误学习） |
| **工作模式** | 常驻，每次会话自动启动，读 memory 恢复上下文 |

### 2.3 前端开发 + UI/UX

| 维度 | 说明 |
|------|------|
| **身份** | 子代理，由 Scrum Master 按需 spawn |
| **核心职责** | 页面开发、组件封装、状态管理、前后端联调、UI/UX 设计 |
| **技术栈** | Vite + React 18 + TypeScript + Ant Design 5 + Zustand |

| **触发条件** | 需要新建/修改前端页面时 spawn |
| **产出交付** | 代码写入 frontend/ 目录 → Scrum Master 审查后提交 |

### 2.4 后端开发

|------|------|

## 三、协作机制

### 3.1 任务流转

```
你提需求
  → Scrum Master 拆解为 Task 列表
  → 按依赖顺序 spawn 子代理执行
  → 子代理完成，代码写入项目目录
  → Scrum Master Code Review（架构+安全+一致性）
  → 审查通过 → Git commit + push
  → 审查不通过 → 反馈给子代理修复（或 Scrum Master 直接修）
  → 所有 Task 完成 → 通知你验收
```

### 3.2 信息共享

| 共享方式 | 用途 | 持久性 |
|----------|------|--------|
| 项目文件系统 | 代码共享，子代理读写同一目录 | ✅ 持久 |
| Git + GitHub | 版本控制，代码审查记录 | ✅ 持久 |
| memory/ 日记文件 | 进度持久化，关机恢复上下文 | ✅ 持久 |
| .learnings/ | 错误学习，跨会话积累 | ✅ 持久 |
| 子代理 task 描述 | 上下文传递，spawn 时注入 | ⚠️ 单次会话 |

### 3.3 安全机制

| 机制 | 频率 | 内容 |
|------|------|------|
| Code Review | 每次提交前 | 架构一致性、代码规范、安全漏洞 |
| 安全 Checklist | 每个 Sprint 末 | 数据加密、权限校验、SQL注入、XSS、CSRF |
| 敏感信息保护 | 持续 | Token/密码不写入代码，.env 不入库 |
| 学习积累 | 持续 | .learnings/ 记录错误与改进，防止重复犯错 |

## 五、Sprint 迭代节奏

### 5.1 Sprint 规则

| 维度 | 规则 |
|------|------|
| **周期** | 1周（按在线时长推进，非自然日） |
| **目标** | 每个 Sprint 交付一个可演示的功能增量 |
| **定义** | "完成" = 代码通过 Review + 编译无错 + 功能可演示 |
| **节奏** | 首日你定目标 → 中间自动推进 → 末日你验收 |

### 5.2 Sprint 看板

```
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│  Backlog │  │  To Do   │  │  Doing   │  │   Done   │
│  产品待办 │  │  Sprint  │  │  开发中  │  │   已完成  │
│          │  │  待办     │  │          │  │          │
│ · 案件管理│  │ · 合同API│  │ · 前端联调│  │ · 后端骨架│
│ · AI接入 │  │ · 数据迁移│  │          │  │ · 前端骨架│
│ · 在线接单│  │          │  │          │  │ · 中间件  │
└──────────┘  └──────────┘  └──────────┘  └──────────┘
```

## 七、运行约束与边界

| 约束 | 说明 |
|------|------|
| 子代理串行执行 | 一次 spawn 一个子代理，任务按依赖顺序串行 |
| 关机即暂停 | PC 关机/断连后所有任务中断，进度已持久化到文件 |
| 开机恢复 | 下次上线 Scrum Master 读 memory 文件恢复上下文，无缝继续 |
| 无自动常驻 | QClaw 非后台服务，需要你发消息或 cron 触发 |
| 代码审查前置 | 所有子代理产出必须经 Scrum Master 审查后才能提交 |
